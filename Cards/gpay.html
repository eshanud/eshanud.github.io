01 The Problem

Same screen. Three completely different realities.

Not "how do we show a better error" — but "how do we design for uncertainty when the frontend and backend have different clocks?"

Safe Failure
No debit — money never left
Mild concern
Red error screen ✕
Pending Reversal
Debited, refund initiated but not confirmed
High anxiety
Same red screen ✕
Delayed / Stuck
Debited, refund delayed or bank silent
Panic, mistrust
Same red screen ✕
Unknown Window
Backend hasn't confirmed yet — 30 sec to 5 min
Worst: user acts on wrong assumption
No app handles this ✕
Usually the system knows. During the Unknown Window it doesn't — and every app pretends otherwise.
We designed for the knowable states. The unknown window is the honest gap.

04 Solution

State-aware failure experience. Not a new screen — a new layer.

Scope honesty — what this actually changes
Ships now · no new infra
Debit status as headline — confirmed vs. unconfirmed
Retry hard-blocked once debit is confirmed
Ownership label — "HDFC Bank is responsible"
Needs bank API · legal review
Real-time 4-stage refund tracker
Proactive push on each state change
Pre-filled dispute on SLA breach — user submits
"Your money is safe" copy — liability risk

One classification upstream. Three completely different user experiences.

Safe Failure
No debit confirmed
"No money was deducted"
Retry →
Pending Reversal
Debit confirmed, reversal initiated
"₹X is safe with your bank"
Track refund →
Delayed Refund
Debit confirmed + timeout exceeded
Timeline + escalation surfaced
Raise dispute →

Retry is blocked when debited. Not a soft warning — a hard gate. Prevents duplicate payment: the #1 real-world harm in the current experience.

Technical reality — the 24-hour Unknown

NPCI doesn't push real-time state to PSPs, so between debit-confirm and the next ACK the system is blind — sometimes 24h. Fallback: past 4h stale, the tracker freezes the step ("No update from HDFC since 9:00 AM") and shows the bank's line. We never invent progress.

Name the responsible party — no app does this

HDFC Bank is processing your refund.
GPay is tracking this for you.

Confidence-gated copy — promise only what the system knows

High
"Expected by Thursday, 28 March" Bank ACK received · inside SLA
Medium
"Usually within 3–5 business days" Reversal pending · bank response delayed
Low
"Taking longer than expected — we're tracking it" SLA window exceeded

The dispute pre-fills but never auto-files — the submit tap stays with the user (RBI compliance). Trust is earned by being checkable: the NPCI ref, the bank's real number, and an honest "no update yet" all live in the screens below.

Final position
The ₹3,200 was never lost. It sat in a bank queue the whole time — the user just couldn't see it.

The real failure wasn't the payment. It was the silence that followed.

GPay can't make refunds faster. It can make them visible. That's the whole redesign.
